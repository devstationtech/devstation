# Reference: Release Procedure & Version Namespaces

## How a release fires

`release.yml` runs on **every push to `main`**. After the quality gate:

1. `deno task release:version` reads `version` from the root `deno.json` and
   computes tag `v<version>`.
2. The release job proceeds **only if that tag does not exist yet** — so
   "bumping the version" is what cuts a release, and a merge without a bump
   no-ops (quality still runs).
3. `deno task build:release` compiles four self-contained binaries
   (`linux-x64`, `darwin-x64`, `darwin-arm64` as `tar.gz`; `windows-x64` as
   `zip`), each embedding the engine, an OpenTofu binary, `templates/`, and
   the `blueprints/` catalog.
4. `release:checksums` + `release:manifest` produce `checksums.txt` and
   `latest.json` (what the curl-pipe installer reads); `gh release create`
   uploads everything; GitHub Pages redeploys the installer at
   `devstation.tech`.

Release notes are a generic autogenerated string — there is no changelog file,
so commit messages are the de-facto history. Keep them release-quality.

## Bumping the product version

- Edit **only** the root `deno.json`. Stable SemVer `X.Y.Z` — the version
  script rejects prereleases and `v`-prefixes.
- Do it in its own PR/commit (`chore: release vX.Y.Z`), never inside a feature
  PR.
- Optionally sync the two dev-mode stubs so from-source runs report the new
  version (the release build overwrites and restores them anyway):
  `server/src/build-info.ts`, `tui/ink/src/cli/version.ts`.
- Local dry-run of the linux artifact chain: `make release`
  (= check + test + `build:linux` + checksums + manifest into `dist/`).

## Anti-drift invariants

- `version` lives in exactly one place (root `deno.json`). Workspace member
  `deno.json` files must **not** carry a `version` field.
- `server/src/build-info.ts` and `tui/ink/src/cli/version.ts` are dev stubs;
  the release build stamps real values and restores the stubs in `finally` —
  never commit a release-stamped stub.
- The MCP server (`server/src/shared/inbound/mcp/server.ts`) imports `VERSION`
  from `build-info.ts` — never hardcode a version literal there.

## The three version namespaces

| Namespace | Where | When to bump |
|---|---|---|
| Product | root `deno.json` | Every release; the only version that ships |
| RPC contract | `libs/jsonrpc-schemas/<bc>.openrpc.json` `info.version` | Only on a **breaking** change to that BC's RPC surface (field removed/renamed). Additive evolution doesn't bump. Regenerate with `deno task contracts:codegen` and commit the `.gen.ts` |
| Blueprint | `blueprints/<name>/blueprint.yaml` `version` | On every behavioral change to the recipe — policy in the `blueprint-dsl` skill (`references/authoring.md`) |

These are independent: a product release does not imply contract or blueprint
bumps, and vice versa.
